# 축약 PRD — 독서 활동 관리 웹 서비스

## 1. 제품 개요

### 1.1 제품명

가칭: **ReadLog**

### 1.2 제품 설명

ReadLog는 사용자가 자신의 독서 활동을 기록하고 관리할 수 있는 개인 독서 관리 웹 서비스다.  
책의 읽기 상태를 관리하고, 독서 노트와 진행률을 기록하며, 태그, 통계, 목표, 캘린더를 통해 독서 습관을 시각적으로 확인할 수 있다.

### 1.3 제품 목표

- 사용자가 독서 기록을 체계적으로 관리할 수 있게 한다.
- 책별 진행 상황과 노트를 함께 저장할 수 있게 한다.
- 검색, 필터, 태그를 통해 원하는 책을 쉽게 찾을 수 있게 한다.
- 통계, 목표, 캘린더를 통해 독서 습관을 추적하고 동기부여를 제공한다.

### 1.4 핵심 기능 범위

- 사용자 인증
- 독서 기록 등록 / 수정 / 삭제
- 내 서재 조회
- 독서 노트 관리
- 독서 진행률 관리
- 검색 / 정렬 / 필터
- 태그 시스템
- 월별 독서 통계
- 독서 목표 설정
- 독서 캘린더

### 1.5 책 정보 입력 정책

- 기본은 사용자가 책 정보를 수동으로 입력한다.
- 추후 **카카오 책 검색 API**를 연동하여 검색 후 책 정보를 가져올 수 있도록 설계한다.

## 2. 기능 요구사항

## 2.1 사용자 인증

- 사용자는 이메일과 비밀번호로 회원가입할 수 있어야 한다.
- 사용자는 로그인 / 로그아웃할 수 있어야 한다.
- 인증된 사용자만 자신의 서재, 노트, 통계, 목표 데이터에 접근할 수 있어야 한다.
- 인증이 필요한 페이지는 비로그인 상태에서 접근 시 로그인 페이지로 이동해야 한다.

### 정책

- 이메일은 중복될 수 없다.
- 비밀번호는 해시 저장해야 한다.
- 사용자 데이터는 본인만 조회/수정/삭제 가능해야 한다.

---

## 2.2 독서 기록 등록

- 사용자는 책을 직접 등록할 수 있어야 한다.
- 사용자는 등록한 책 정보를 수정할 수 있어야 한다.
- 사용자는 등록한 책을 삭제할 수 있어야 한다.

### 책 정보 항목

- 제목 (필수)
- 저자
- 출판사
- 표지 이미지
- 총 페이지 수
- 읽기 상태 (`읽고 싶음`, `읽는 중`, `완독`)
- 시작일
- 완독일
- 한줄평
- 평점

### 정책

- 제목은 필수다.
- 평점은 **5점 만점, 0.5 단위**로만 입력 가능해야 한다.
- 총 페이지 수는 0보다 커야 한다.
- 완독 상태일 경우 완독일을 입력할 수 있어야 한다.
- 책 삭제 방식은 **Hard Delete**를 적용한다.

---

## 2.3 내 서재

- 사용자는 자신의 책 목록을 조회할 수 있어야 한다.
- 사용자는 상태별로 책을 나눠서 볼 수 있어야 한다.
    - 전체
    - 읽고 싶음
    - 읽는 중
    - 완독
- 각 책 항목에는 최소 다음 정보가 보여야 한다.
    - 표지
    - 제목
    - 저자
    - 상태
    - 진행률 또는 평점
- 사용자는 목록에서 책 상세 페이지로 이동할 수 있어야 한다.

### 정책

- 목록은 로그인한 사용자 기준으로만 조회된다.
- 기본 정렬 기준은 하나 정해서 제공해야 한다.
- 목록이 많아질 경우 페이지네이션 또는 무한 스크롤이 필요하다.

---

## 2.4 독서 노트

- 사용자는 특정 책에 대해 노트를 작성할 수 있어야 한다.
- 사용자는 노트 목록을 조회할 수 있어야 한다.
- 사용자는 노트를 수정할 수 있어야 한다.
- 사용자는 노트를 삭제할 수 있어야 한다.

### 노트 항목

- 노트 내용 (필수)
- 인상 깊은 문장
- 페이지 번호
- 챕터명 또는 위치 정보

### 정책

- 노트는 반드시 특정 책에 속해야 한다.
- 내용 없는 빈 노트는 저장할 수 없다.
- 페이지 번호는 1 이상의 숫자만 허용한다.

---

## 2.5 독서 진행률

- 사용자는 책별 현재 읽은 페이지 수를 입력할 수 있어야 한다.
- 시스템은 총 페이지 수를 기준으로 진행률을 계산해 표시해야 한다.
- 진행률은 책 목록과 상세 페이지 모두에서 확인 가능해야 한다.
- 사용자는 책 상태를 `읽는 중` 또는 `완독`으로 변경할 수 있어야 한다.

### 계산 규칙

- 진행률 = 현재 읽은 페이지 수 / 총 페이지 수 × 100
- 총 페이지 수가 없는 경우 진행률은 계산하지 않는다.
- 완독 상태인 경우 진행률은 100%로 처리할 수 있다.

### 정책

- 진행률은 **현재 읽은 페이지 수 기준**으로 처리한다.
- 현재 읽은 페이지 수는 0 이상이어야 한다.
- 현재 읽은 페이지 수는 총 페이지 수를 초과할 수 없다.

---

## 2.6 검색 / 정렬 / 필터

- 사용자는 제목으로 책을 검색할 수 있어야 한다.
- 사용자는 저자로 책을 검색할 수 있어야 한다.
- 사용자는 상태별 필터를 적용할 수 있어야 한다.
- 사용자는 태그별 필터를 적용할 수 있어야 한다.
- 사용자는 정렬 기준을 선택할 수 있어야 한다.

### 정렬 기준 예시

- 최근 등록순
- 최근 수정순
- 평점순
- 완독일순
- 제목순

### 정책

- 검색은 부분 일치를 지원해야 한다.
- 검색, 필터, 정렬은 조합해서 사용할 수 있어야 한다.
- 정렬은 한 번에 하나의 기준만 적용한다.

---

## 2.7 태그 시스템

- 사용자는 태그를 생성할 수 있어야 한다.
- 사용자는 책에 여러 개의 태그를 연결할 수 있어야 한다.
- 사용자는 책에 연결된 태그를 제거할 수 있어야 한다.
- 사용자는 태그 기준으로 책을 필터링할 수 있어야 한다.

### 정책

- 태그는 사용자 단위로 관리한다.
- 태그는 **색상(color)** 을 지원한다.
- 태그는 **아이콘을 지원하지 않는다**.
- 태그명은 사용자 단위에서 중복되지 않도록 관리한다.
- 태그명 중복 비교 시 **대소문자를 구분하지 않는다**.
- 책과 태그는 다대다 관계다.

---

## 2.8 월별 독서 통계

- 사용자는 특정 월의 독서 통계를 조회할 수 있어야 한다.
- 기본 조회 기준은 현재 월이어야 한다.
- 월을 변경하면 해당 월의 통계가 갱신되어야 한다.

### 제공 지표

- 이번 달 읽은 책 수
- 이번 달 완독한 책 수
- 이번 달 읽은 페이지 수
- 평균 평점
- 태그별 독서 비율

### 정책

- 월별 통계는 **완독 통계와 활동 발생 통계를 모두 기록**해야 한다.
- 데이터가 없는 월은 0 또는 빈 상태 UI로 표시한다.

### 통계 기준

- **완독 통계**: 해당 월에 완독일이 존재하는 책 기준
- **활동 발생 통계**: 해당 월에 페이지 기록이 존재하는 책 또는 독서 활동 기준

---

## 2.9 독서 목표 설정

- 사용자는 독서 목표를 생성할 수 있어야 한다.
- 사용자는 생성한 목표를 수정할 수 있어야 한다.
- 시스템은 실제 독서 데이터와 비교하여 목표 달성률을 계산해야 한다.
- 목표 달성 여부를 상태로 표시해야 한다.

### 목표 예시

- 올해 N권 읽기
- 이번 달 N페이지 읽기
- 이번 달 N일 독서하기

### 정책

- **월간/연간 목표를 모두 지원**해야 한다.
- 목표는 기간 단위와 측정 기준을 함께 가져야 한다.
- **동일 기간, 동일 타입 목표는 하나만 허용**한다.

---

## 2.10 독서 캘린더

- 사용자는 월 단위 독서 캘린더를 조회할 수 있어야 한다.
- 날짜별 독서 활동 여부를 확인할 수 있어야 한다.
- 날짜를 클릭하면 해당 날짜의 독서 활동 상세를 확인할 수 있어야 한다.
- 활동량에 따라 시각적으로 차등 표시할 수 있어야 한다.

### 정책

- 캘린더는 **페이지 기록만으로 채운다**.
- 날짜별 독서 활동 기록은 페이지 수 기준으로 관리한다.
- 초기 버전에서는 페이지 기록 중심으로 단순하게 구현한다.

---

## 3. 비기능 요구사항

### 3.1 성능

- 서재 목록과 주요 화면은 체감상 빠르게 로드되어야 한다.
- 검색 / 필터 / 정렬은 과도한 지연 없이 동작해야 한다.
- 통계 및 캘린더 데이터 조회는 월 단위 기준에서 무리 없이 응답해야 한다.

### 3.2 보안

- 비밀번호는 평문 저장을 금지한다.
- 인증이 필요한 API는 서버에서 사용자 권한 검증을 수행해야 한다.
- 사용자는 자신의 데이터에만 접근 가능해야 한다.

### 3.3 사용성

- 책 등록, 수정, 노트 작성 흐름은 직관적이어야 한다.
- 빈 목록, 빈 통계, 검색 결과 없음에 대한 빈 상태 UI가 필요하다.
- 모바일과 데스크톱에서 모두 사용할 수 있도록 반응형 UI를 고려한다.

### 3.4 데이터 무결성

- 잘못된 페이지 수, 평점, 날짜 입력은 저장되면 안 된다.
- 책 삭제 시 관련 데이터는 Hard Delete 정책에 맞게 일관되게 제거되어야 한다.
- 통계/목표/캘린더 데이터 기준이 서로 충돌하지 않아야 한다.

### 3.5 확장성

- 추후 카카오 책 검색 API 연동이 가능하도록 책 정보 구조를 유연하게 설계한다.
- 소셜 기능이나 공개 서재 기능을 붙일 수 있도록 사용자 데이터 구조를 분리한다.

---

## 4. 확정 정책

### 4.1 책 정보 입력 방식

- 기본은 수동 입력
- 추후 카카오 책 검색 API 연동 예정

### 4.2 독서 활동 기록 단위

- 페이지 수 기준으로 기록

### 4.3 월별 통계 기준

- 완독 통계와 활동 발생 통계를 모두 지원

### 4.4 목표 지원 범위

- 월간 목표 지원
- 연간 목표 지원

### 4.5 목표 중복 정책

- 동일 기간, 동일 타입 목표는 하나만 허용

### 4.6 책 삭제 방식

- Hard Delete

### 4.7 진행률 계산 기준

- 현재 읽은 페이지 수 기준

### 4.8 캘린더 활동 기준

- 페이지 기록만으로 캘린더 구성

### 4.9 태그 정책

- 색상 지원
- 아이콘 미지원
- 태그명 비교 시 대소문자 구분 없음

### 4.10 평점 정책

- 5점 만점
- 0.5 단위 허용
